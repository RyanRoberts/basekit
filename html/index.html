Installation ↑

  1. Clone and then copy to or download to your project.
  2. $ npm install at the project/basekit root
  3. $ gulp to start compiling CSS changes
  4. Configure BaseKit from /css/settings/
  5. Add new .scss files to the root of the /css directory and import them at the bottom of basekit.scss
  6. Additional config may be required in the root /css files (such as _colors.scss)

Fonts ↑

Responsive, viewport based, fonts should be restricted to use on headings and isolated blocks of text. Avoid using it on copy paragraphs and long blocks of text. I would also advise against using medium or large sizes nearby normal (non-responsive) blocks of text because these viewport based texts can actually end up smaller at certain screen sizes.

I've never been a fan of styling h1, h2, h3, etc directly or by tying their size and style to their heading number. What if h2 and h4 should visually look the same but not in every instance? Over time and with multiple maintainers the CSS and HTML end up tied in knots.

The solution is simple; just define font styles to utility classes and apply where required regardless of the markup. If you want to style one list item to look like some hero text, go right ahead! It does not matter, the semantics are in your HTML markup and content.

Breakpoints & Grid ↑

Easy to use inline-block based grid system, utilising Sass MQ.

The grid system CSS actually lives in its own repo called basegrid. This will soon get pulled back into BaseKit in order to simplify maintenance and setup.

Below is an example of my most commonly used breakpoint configuration, it provides a nice balance of control and code. For each breakpoint, including the mobile first base level, the code is repeated so more breakpoints equals more code that may not be used.

Breakpoint Viewport width Device Sass Usage BaseGrid Usage
n/a n/a n/a n/a data-bk="6"
palm 600px + Horizontal small phone or large phone mq(palm){…} data-bk="palm:6"
hands 900px + Portrait tablet type device mq(hands){…} data-bk="hands:6"
lap 1200px + Landscape tablets and small laptops mq(lap){…} data-bk="lap:6"
desk 1800px + Larger screens mq(desk){…} data-bk="desk:6"

The following is a more paired back example which not only reduces the CSS produced but also offers less control over layout break points. This is useful for lower cost projects because it forces fewer decisions through simplicity, or something like that.

Breakpoint Viewport width Device Sass Usage BaseGrid Usage
n/a n/a n/a n/a data-bk="6"
s 600px + Handheld smaller devices mq(s){…} data-bk="s:6"
m 900px + Tablets and mega-phones mq(m){…} data-bk="m:6"
l 1200px + Larger screens mq(l){…} data-bk="l:6"

Class Naming ↑

BaseKit makes use of two primary types of class; the utility class and the BEM class.

Utility classes aim to be reuseable and single purpose, unlike BEM classes they should not have sub elements but they can have modifiers.

BEM classes are most likely to be used in unique project modules or components and not within the core CSS files. If a component of the sites lends well to the Block Element Modifier convention (such as tabbed content) then you're advised to use this.

  .tabs {…}
  .tabs__nav {…}
  .tabs__nav-item {…}
  .tabs__content {…}
  .tabs__content--active {…}

In brief, when creating classes take the following approach:

  1. Describe the thing you're creating or doing: .site-logo
  2. If the thing has different states do that via a modifier to the class: .site-logo--inverted
  3. If it has elements then make use of the BEM method.
  4. Hyphenate words and names within classes. Don't use camelCase or under_scores (unless it's a BEM element of course)
  5. Aim for single purposes utility classes first, BEM second, project specific last
  6. Do not worry about using lots of classes in your HTML
  7. Do not worry about describing your content using classes, this is the purpose of HTML

Links & Navigation ↑

For years the traditional mathod of styling links would inevitably over time drive me insane with specificity problems and overrides. So eventually I came to the conclusion that links should simply be scoped, via classes.

This does mean that you have to be sure to catch every link in your templates, otherwise they'll appear unstyled. It also means you'll be using unsemantic blah blah blah classes. However it does produce easier to maintain code with fewer long-term headaches. Just create a link style on a class and stick it where it needs to go in your markup (be sensible!).
