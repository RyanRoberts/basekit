Typography

Responsive, viewport based, fonts should be restricted to use on headings and isolated blocks of text. Avoid using it on copy paragraphs and long blocks of text.

I would also advise against using medium or large sizes nearby normal (non-responsive) blocks of text because these viewport based texts can actually end up smaller at certain screen sizes.

I've never been a fan of styling h1, h2, h3, etc directly or by tying their size and style to their heading number. What if h2 and h4 should visually look the same but not in every instance? Over time and with multiple maintainers the CSS and HTML end up tied in knots.

The solution is simple; just define font styles to utility classes and apply where required regardless of the markup. If you want to style one list item to look like some hero text, go right ahead! It does not matter, the semantics are in your HTML markup and content.

Default sans-serif

-apple-system, BlinkMacSystemFont, 'Segoe UI', 'Helvetica Neue', 'Helvetica', 'Roboto', 'Oxygen', 'Ubuntu', 'Cantarell', 'Fira Sans', 'Droid Sans', 'arial', sans-serif

Default serif

georgia, 'nimbus roman no9 l', freeserif, times, 'times new roman', serif

Default monospace

'SFMono-Regular', Consolas, 'Liberation Mono', Menlo, Courier, monospace

Font Sizes

In Basekit all headings are set to the same size by default, since it is not necessarily true that you would want a large h1 and a small h4. Sometimes you might want them to be the same size, othertimes it may be have visual weight that does not consist with semantic weight.

Due to this, Basekit abstracts out font weights and styles to specific utility classes which must be applied to get the desired style. Otherwise all headings and text sizes will, for the most part, remain the same. This also has the added advantage of being able to apply consistent styles to any text elements, regardless of the markup.
